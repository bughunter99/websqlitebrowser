websqlitebrowser metadata/skill 작성 가이드

목적
- Chat이 SQLite 스키마/샘플 데이터만으로 애매한 질문을 받을 때, 비즈니스 의미를 보강하기 위한 문서 규칙이다.
- 이 가이드대로 작성하면 Chat의 Context/Trace에 문서가 안정적으로 포함된다.

적용 위치
- repository/metadata/tables/*.md
- repository/metadata/databases/*.md
- repository/metadata/skills/*.md

로드 우선순위(현재 구현 기준)
1) databases/{db명}.md
2) skills/{db명}-*.md (DB 전용 스킬)
3) skills/skill*.md (전역 스킬)
4) tables/{테이블명}.md (질문에 언급된 테이블 우선)

파일명 규칙
1) 테이블 문서
- 경로: repository/metadata/tables/{table_name}.md
- 예: repository/metadata/tables/customers.md

2) DB 문서
- 경로: repository/metadata/databases/{db_stem}.md
- 예: sample.db -> repository/metadata/databases/sample.md

3) 스킬 문서
- 전역: repository/metadata/skills/skill01.md, skill02.md ...
- DB 전용: repository/metadata/skills/{db_stem}-skill01.md, {db_stem}-skill02.md ...
- 예: repository/metadata/skills/sample-skill01.md

작성 원칙
- 짧고 단정하게: 한 문단 2~4문장, 목록 중심.
- 사실/정책/용어를 분리해서 작성.
- SQL 힌트는 읽기 전용(SELECT/WITH/PRAGMA/EXPLAIN)만 작성.
- 추정/가정은 반드시 명시: "추정", "확인 필요".
- 값 범위, 단위, 시간대, 기준일을 명확히 적는다.
- 길이 권장: 파일당 300~1500자.

권장 섹션 스키마(중요)
- 아래 3가지는 꼭 넣는다.
1) 속성 의미: 컬럼/지표/코드값의 비즈니스 의미
2) 질문 유형: 실제로 자주 들어오는 질문 패턴
3) 쿼리 전략: 질문 유형별 집계/필터/조인 방식

권장 헤더 이름(LLM이 읽기 쉽게 통일)
- ## 속성 의미
- ## 질문 유형
- ## 쿼리 전략
- ## SQL 템플릿
- ## 주의사항

# 테이블: customers

## 1) 한 줄 정의
- 고객 마스터. 가입 시점 기준의 기본 식별/거주 정보.

## 2) 컬럼 의미
- id: 고객 PK. 시스템 내부 식별자.
- name: 고객명(표시용).
- city: 기본 거주 도시.
- joined_at: 가입일시(기준 타임존: KST, 포맷: YYYY-MM-DD 또는 ISO8601).

## 질문 유형
- 고객 수: "고객이 몇 명이야?", "전체 고객 수 알려줘"
- 지역별 고객 수: "도시별 고객 수"
- 기간 유입: "이번 달 가입 고객 수"

## 쿼리 전략
- 고객 수: COUNT(DISTINCT id)
- 기간 조건 없음: 전체 기간
- 기간 조건 있음: joined_at을 날짜로 변환 후 필터
- 지역 집계: city 기준 GROUP BY

## 3) 해석 규칙
- "고객 수"는 기본적으로 COUNT(DISTINCT id).
- 기간 조건이 없으면 전체 기간으로 해석.
- 탈퇴 개념 컬럼이 없으면 활성/비활성 구분 없이 집계.

## 4) 주의사항
- name은 중복 가능하므로 식별 키로 사용하지 않음.
- joined_at 문자열 포맷 불일치 가능성 있음(질문 시 명시).

## 5) 자주 쓰는 조회 예시
- 전체 고객 수
  SELECT COUNT(DISTINCT id) AS customer_count FROM customers;
- 도시별 고객 수
  SELECT city, COUNT(DISTINCT id) AS customer_count
  FROM customers
  GROUP BY city
  ORDER BY customer_count DESC;

# 데이터베이스: sample

## 1) 용도
- 데모/검증용 소규모 주문 데이터셋.

## 2) 핵심 테이블
- customers: 고객 기본 정보
- orders: 주문 헤더/금액
- sample: 테스트용 보조 데이터

## 3) 공통 정책
- 금액 단위: KRW
- 기간 기본값: 전체 기간
- 날짜 해석: joined_at/order_date가 문자열일 수 있어 형 변환 필요

## 4) 답변 정책
- 수치 답변 시 기준 테이블/조건을 문장에 함께 명시.
- 애매한 질문은 대상 테이블을 먼저 특정하고 진행.

# 전역 스킬 01: 집계 질문 해석

## 규칙
- "몇 명/몇 건/얼마" 질문은 먼저 측정대상(엔티티)을 확정한다.
- 엔티티 미지정 시 customers, orders 순으로 후보 제시 후 기본값을 명시한다.
- 결과 문장에는 반드시 기준(테이블/기간/조건)을 포함한다.

## 질문 유형
- 수량형: "몇 명/몇 건/얼마"
- 비교형: "A가 B보다 많은가"
- 추세형: "월별/주별 변화"

## 쿼리 전략
- 수량형: COUNT/SUM 우선, 기준 키를 명시
- 비교형: 동일 기간/조건으로 두 집계를 계산
- 추세형: strftime('%Y-%m', date_col) 등 동일 버킷으로 집계

## 출력 스타일
- 첫 문장: 결론(숫자)
- 둘째 문장: 기준
- 필요 시 셋째 문장: 한계/가정

# sample DB 스킬 01: 주문 분석

## 규칙
- "매출"은 orders.total_amount 합계를 의미.
- 주문 건수는 COUNT(*) 또는 COUNT(DISTINCT order_id) 중 스키마에 맞는 키 사용.
- 고객 기준 지표는 customers.id를 기본 키로 사용.

## 질문 유형
- "이번 달 매출", "전월 대비 매출"
- "고객당 평균 주문 금액"

## 쿼리 전략
- 기간 비교: 동일 집계식 + 기간 필터만 변경
- 평균 객단가: SUM(total_amount) / COUNT(DISTINCT customer_id)
- null 방어: COALESCE 사용

## SQL 힌트
- 월별 매출은 strftime('%Y-%m', order_date) 기준으로 집계.

작성 체크리스트

- [ ] 질문에서 쓰는 도메인 용어 정의가 있는가?
- [ ] "질문 유형" 섹션에 실제 질문 문장을 넣었는가?
- [ ] "쿼리 전략" 섹션에 집계/필터/조인 규칙을 넣었는가?
- [ ] 기본 집계 규칙(COUNT 기준, 기간 기본값)이 있는가?
- [ ] 단위/시간대/포맷이 명시되어 있는가?
- [ ] 읽기 전용 SQL 예시만 포함했는가?
- [ ] 과도한 장문 설명 없이 핵심만 적었는가?

좋은 예/나쁜 예
- 나쁜 예: "고객은 중요하다. 상황에 따라 다르다."
- 좋은 예: "고객 수는 customers.id 기준 COUNT(DISTINCT id). 기간 미지정 시 전체 기간 적용."

빠른 시작(5분)
1) tables/customers.md, tables/orders.md부터 작성
2) databases/{db}.md에 단위/기간 기본값 정의
3) skills/skill01.md에 공통 해석 규칙 3~5개 작성
4) DB별로 {db}-skill01.md 1개 추가
5) Chat에서 질문 후 Context/Trace로 문서 로딩 여부 확인

운영 팁
- 문서 변경 후 브라우저 새로고침(Ctrl+F5)하면 UI 캐시 이슈를 줄일 수 있다.
- 답변이 기대와 다르면, 먼저 tables/*.md의 "해석 규칙"을 짧고 강하게 수정한다.
